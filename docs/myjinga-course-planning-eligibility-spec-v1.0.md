# 마진가(Myjinga) 과목 설계 추천 가능 여부 검사 스펙

- 문서 버전: v1.0
- 기준 문서: `docs/myjinga-development-spec-v1.1.md`, `docs/myjinga-course-recommendation-rules-spec-v1.0.md`, `docs/myjinga-curriculum-reference-data-spec-v1.0.md`, `docs/myjinga-project-rules-v1.0.md`, `data/reference/curriculum/review/graduation-requirements-draft.json`
- 작성일: 2026-04-10
- 상태: Draft (추천 제약 로직 정의)
- 적용 범위: 고등학교 보통 교과 기준 선택과목 설계 시 이수 가능 여부 검증, 추천 필터링, 실패/경고 사유 코드 정의
- 영향 범위: 추천 API, 과목 설계 UI, 참조 데이터 구조, 검증 로직 테스트, 관련 개발 문서

---

## 1. 목적

본 문서는 고교학점제 선택과목 추천에서 `추천 점수 계산`보다 먼저 적용되어야 하는 `이수 가능 여부 검사` 기준을 정의한다. 상위 규칙 체계는 `docs/myjinga-course-recommendation-rules-spec-v1.0.md`를 따르며, 본 문서는 그중 `이수 가능 여부 검사`를 분리한 상세 문서다.

이 문서의 목적은 아래 4가지다.

1. 추천 결과가 졸업 이수 규정을 위반하지 않도록 하드 제약을 명시한다.
2. `과목 메타데이터(subjects)`와 `이수 규정(graduationRequirements)`을 분리하여 관리 기준을 고정한다.
3. 추천 API와 UI가 동일한 판정 결과와 동일한 실패 사유 코드를 사용하도록 한다.
4. 아직 전 과목 메타데이터를 채우지 않은 교과군에 대해서도 `최소 검증`이 가능하도록 단계적 검증 정책을 정의한다.
5. 목표 학과 권장 과목을 먼저 배정한 뒤 남은 학점 구간을 검증하는 흐름에서, `이수 가능 여부 검사`의 역할을 고정한다.

## 2. 범위와 전제

### 2.1 현재 범위

1. 대상 학교급: `고등학교 보통 교과`
2. 대상 입학생: `2025학년도 입학생`
3. 적용 교육과정: `2022 개정 교육과정`
4. 규정 출처: `고등학교 보통 교과 교과목 구성` 및 `총 이수 학점` 표

### 2.2 현재 검증 단위

1. `fullGraduationPlan`
  - 3개 학년 전체를 아우르는 최종 이수 계획 검증
2. `recommendationFilter`
  - 추천 후보 과목 세트가 현재 규정상 명백히 불가능한지 사전 필터링

### 2.3 단계적 운영 정책

1. `국어`, `수학`, `영어`, `사회(역사/도덕 포함)`, `과학`은 과목 단위까지 검증한다.
2. `체육`, `예술`, `기술·가정/정보/제2외국어/한문/교양`은 현재 단계에서 `필수 학점 충족 여부`까지만 검증한다.
3. 학교별 실제 개설 과목 여부는 현재 범위에서 제외한다.
4. 학기별 시간표 충돌, 선수 과목, 반 편성 충돌은 현재 범위에서 제외한다.
5. `credit-only-subject-group-requirements-draft.json`은 아직 상세 과목 메타데이터를 채우지 않은 교과군의 임시 기준 데이터로 사용한다.
6. `creditOperationPolicies` 및 학교별 실제 편성 범위 검사는 현재 범위에서 제외한다.
7. 목표 학과 권장 과목(`lockedRequiredSubjects`)은 이미 계획에 포함된 `필수 고정 과목` 입력으로 들어온다고 가정하고 검증한다.

## 4. 입력 데이터 모델

### 4.1 참조 데이터

1. `subjects`
  - 과목명, 교과군, 선택 유형, 학점/평가 정보, 관련 학과/직업
2. `graduationRequirements`
  - 총 이수 학점
  - 필수 이수 학점
  - 자율 이수 학점
  - 창의적 체험활동 학점/시간
  - 교과군별 필수 학점
  - 공통과목 목록
  - 세부 버킷 규칙(예: 한국사/통합사회)
3. `creditOnlySubjectGroupRequirements`
  - 체육, 예술, 기술·가정/정보/제2외국어/한문/교양의 학점 중심 임시 기준 데이터
4. `creditOperationPolicies`
  - 과목 기본 학점과 편성·운영 가능 범위, 예외 학교 유형 정보

### 4.2 사용자 계획 입력

검사 로직은 아래 구조를 입력으로 받는 것을 기본으로 한다.

```json
{
  "schoolType": "general-high-school",
  "entryYear": 2025,
  "targetSchool": "서울예시고",
  "targetMajor": "경영학과",
  "lockedRequiredSubjectNames": ["경제 수학", "경제", "국제 관계의 이해"],
  "completedSubjectNames": ["공통국어1", "공통국어2"],
  "plannedSubjectNames": ["화법과 언어", "주제 탐구 독서"],
  "completedCreditsByGroup": {
    "국어": 8,
    "수학": 4
  },
  "plannedCreditsByGroup": {
    "국어": 6,
    "사회(역사/도덕 포함)": 4
  }
}
```

### 4.3 내부 파생 값

검사 과정에서는 아래 파생 값을 계산한다.

1. `allSubjectNames`
  - 완료 + 계획 과목 합집합
2. `creditsByGroup`
  - 완료 + 계획 학점 합산 결과
3. `commonSubjectCoverage`
  - 공통과목 이수 여부
4. `bucketCoverage`
  - 예: `한국사 6학점`, `통합사회 8학점`

## 5. 검사 단계

### 5.1 1단계: 입력 무결성 검사

1. 과목명이 참조 데이터에 존재하는지 확인
2. 중복 과목 입력 여부 확인
3. 지원하지 않는 학교 유형/입학생 연도 여부 확인
4. 학점 값이 음수이거나 비정상 값인지 확인
5. `lockedRequiredSubjectNames`가 참조 데이터에 존재하는지 확인
6. `lockedRequiredSubjectNames`는 일반 추천 후보가 아니라 `필수 고정 과목`이라는 전제를 확인

### 5.2 2단계: 공통과목 충족 검사

1. `국어`, `수학`, `영어`, `사회`, `과학`의 공통과목 이수 여부 검사
2. `사회(역사/도덕 포함)`은 아래 두 버킷을 별도로 검사
  - `한국사1`, `한국사2`
  - `통합사회1`, `통합사회2`
3. 공통과목이 부족하면 `ineligible`가 아니라 우선 `needsReview`를 반환할 수 있다.
  - 이유: 현재 계획이 중간 상태일 수 있기 때문

### 5.3 3단계: 교과군별 필수 학점 검사

1. `국어 8`, `수학 8`, `영어 8`
2. `사회 14` (`한국사 6 + 통합사회 8`)
3. `과학 10`
4. `체육 10`
5. `예술 10`
6. `기술·가정/정보/제2외국어/한문/교양 16`

### 5.4 4단계: 총 이수 학점 검사

1. `필수 이수 학점 = 84` 충족 여부
2. `자율 이수 학점 = 90` 충족 여부
3. `총 이수 학점 = 192` 충족 여부
4. `창의적 체험활동 = 18학점(288시간)`은 별도 항목으로 표시하되 교과 학점과 분리 계산

### 5.5 5단계: 추천 후보 필터링

1. 하드 제약을 깨는 과목 세트는 추천 후보에서 제외
2. 부족한 공통과목/필수 학점이 있으면 보완 우선순위를 앞세운다.
3. `lockedRequiredSubjectNames`는 일반 추천 후보보다 우선 유지한다.
4. `lockedRequiredSubjectNames`는 충돌이 발생해도 자동 제거하지 않는다.
5. 선택 과목 점수 계산은 위 단계 통과 후 수행한다.

## 6. 판정 상태

### 6.1 `eligible`

아래를 모두 만족하면 `eligible`이다.

1. 입력 무결성 오류 없음
2. 필수 공통과목 충족
3. 교과군별 필수 학점 충족
4. 총 이수 학점 구조 충족

### 6.2 `needsReview`

아래 중 하나면 `needsReview`이다.

1. 아직 전체 3개 학년 계획이 완성되지 않음
2. 일부 교과군은 학점만 있고 과목 세부 매핑이 부족함
3. 공통과목 또는 필수 학점이 아직 부족하지만 미래 계획으로 보완 가능함
4. 참조 데이터가 아직 완전하지 않아 사람 검토가 필요한 경우
5. 목표 학과 필수 배정 과목과 학교 운영 현실 간 충돌이 있는 경우

### 6.3 `ineligible`

아래 중 하나면 `ineligible`이다.

1. 지원하지 않는 학교 유형/연도
2. 참조 데이터에 없는 과목을 핵심 입력으로 사용
3. 동일 과목 중복 등 구조적 오류
4. 현재 계획 구조상 필수 학점 충족이 불가능한 경우

## 9. 현재 단계의 단순화 규칙

1. `체육`, `예술`, `기술·가정/정보/제2외국어/한문/교양`은 과목 메타데이터가 아직 없더라도 `필수 학점 충족 여부`만 검사한다.
2. 위 교과군은 과목명 누락 때문에 실패시키지 않고 `GRAD-SIMPLIFIED-001` 경고를 반환한다.
3. 현재 단계에서 추천 엔진은 `국수영사과` 중심으로 과목 추천을 계산한다.
4. `graduationRequirements.subjectGroupRequirements[].validationMode = "credits-only"`이면 세부 과목 일치 대신 학점 총량만 본다.

## 10. API 및 UI 적용 원칙

### 10.1 API 원칙

1. 추천 API는 점수 계산 전 `eligibility check`를 먼저 수행한다.
2. 검사 결과가 `ineligible`이면 추천 점수 결과를 최종 추천으로 확정하지 않는다.
3. `needsReview`이면 추천 결과는 보여주되, 보완이 필요한 규칙 코드를 함께 반환한다.

### 10.2 UI 원칙

1. 사용자에게는 `추천 과목`과 함께 `부족한 필수 이수 조건`을 같이 보여준다.
2. `needsReview` 상태에서는 `추천 가능하지만 이수 조건 보완 필요` 문구를 노출한다.
3. `ineligible` 상태에서는 대체 과목 제안보다 먼저 규정 위반 사유를 보여준다.

4. 선수 과목/권장 이수 순서 반영
5. 학과별 요구 교과 조합과 졸업 규정을 동시에 만족시키는 최적화 로직

## 12. 변경 이력

### v1.0 - 2026-04-10

1. 이수 규정 기반 추천 가능 여부 검사 스펙 신규 작성
2. `graduationRequirements`를 추천 제약의 기준 데이터로 정의
3. 현재 단계의 `학점 중심 단순화 규칙`과 판정 상태/규칙 코드 정의
4. `creditOperationPolicies`를 후속 학점 편성 범위 검증 입력으로 추가
5. 통합 상위 문서(`myjinga-course-recommendation-rules-spec-v1.0.md`)를 기준 문서로 연결
6. 목표 학과 우선 배정 과목(`lockedRequiredSubjects`)을 `필수 고정 과목`으로 검증해야 한다는 책임을 명시
