# 마진가(Myjinga) 프로젝트 규칙서

- 문서 버전: v1.0
- 작성일: 2026-03-05
- 적용 범위: 기획, 설계, 구현, 테스트, 배포, 운영, 문서화 전 단계
- 목적: 1인/소규모 개발 환경에서도 재현 가능한 품질 기준 유지

---

## 1. 운영 원칙

1. 사용자 데이터 안전이 기능 속도보다 우선이다.
2. 문서 없는 결정은 임시 결정으로 간주한다.
3. 배포 가능한 단위로 작게 만들고, 빠르게 검증한다.
4. 외부 API 의존 기능은 실패를 기본 가정으로 설계한다.

---

## 2. 문서 체계(Documentation System)

## 2.1 문서 분류

1. Product 문서: 요구사항, 정책, 사용자 시나리오
2. Engineering 문서: 아키텍처, API, 데이터, 보안, 테스트
3. Operations 문서: 배포, 모니터링, 장애 대응, 백업/복구

## 2.2 필수 문서 목록

1. 고도화 개발 실행 명세서 (`myjinga-development-spec-*.md`)
2. 프로젝트 규칙서 (`myjinga-project-rules-*.md`)
3. API 명세서 (OpenAPI 또는 markdown)
4. Firestore 규칙/인덱스 문서
5. QA 시나리오 및 릴리스 체크리스트
6. 장애 대응 런북

## 2.3 문서 네이밍 규칙

1. 파일명 형식: `myjinga-{doc-type}-v{major.minor}.md`
2. 버전 증가 기준:
  - major: 정책/구조가 호환되지 않게 변경
  - minor: 호환 가능한 내용 추가/정교화
3. 문서 헤더에 `작성일`, `상태`, `적용 범위`를 반드시 기록

## 2.4 문서 변경 워크플로우

1. 변경 목적 정의
2. 영향 범위 표기(기능/API/데이터/테스트)
3. 관련 문서 동시 수정
4. 변경 이력(Changelog) 기록

---

## 3. 요구사항 관리 규칙

## 3.1 요구사항 식별자

모든 요구사항에 ID를 부여한다.

- 형식: `REQ-{도메인}-{번호}`
- 예시: `REQ-AUTH-001`, `REQ-AI-004`

## 3.2 추적성(Traceability)

요구사항은 아래 항목과 1:1 또는 1:N으로 연결되어야 한다.

1. API 계약 ID
2. DB 필드/컬렉션
3. 테스트 케이스 ID
4. 릴리스 노트 항목

요구사항과 구현/테스트 연결이 없는 변경은 머지 금지.

---

## 4. 기술 의사결정 규칙(ADR)

## 4.1 ADR 작성 조건

아래 변경은 반드시 ADR(Architecture Decision Record) 작성:

1. 외부 서비스 교체(OpenAI, PG사, 인증 등)
2. DB 스키마 핵심 변경
3. 비용 구조에 영향이 큰 변경
4. 보안/개인정보 처리 정책 변경

## 4.2 ADR 최소 항목

1. 배경(Context)
2. 결정(Decision)
3. 대안(Alternatives)
4. 영향(Consequences)
5. 롤백 조건

---

## 5. 코드/브랜치/커밋 규칙

## 5.1 브랜치 전략

1. `main`: 배포 가능한 상태만 유지
2. `feat/*`: 기능 개발
3. `fix/*`: 버그 수정
4. `chore/*`: 비기능 작업

## 5.2 커밋 규칙

Conventional Commits 사용:

1. `feat: ...`
2. `fix: ...`
3. `refactor: ...`
4. `docs: ...`
5. `test: ...`
6. `chore: ...`

## 5.3 PR 규칙

PR에는 아래를 반드시 포함:

1. 목적/배경
2. 변경 요약
3. 테스트 결과
4. 위험/롤백 방법
5. 관련 요구사항 ID

---

## 6. 구현 규칙

## 6.1 Next.js/TypeScript 규칙

1. TypeScript `strict` 유지
2. `any` 사용 금지(불가피한 경우 근거 주석 필수)
3. 서버 비밀키는 server route 내부에서만 접근
4. 비즈니스 로직은 `lib/`에 분리하고 UI 컴포넌트는 표현 책임만 갖는다

## 6.2 API 규칙

1. 모든 API는 입력 검증(zod 등) 필수
2. 표준 오류 코드 체계 유지
3. 멱등성이 필요한 API는 idempotency key 설계 필수
4. 응답 스키마는 문서와 실제 구현이 일치해야 한다

## 6.3 Firestore 규칙

1. 컬렉션/필드 네이밍은 `camelCase` 통일
2. 쓰기 시 `createdAt`, `updatedAt` 자동 기록
3. 쿼리 추가 전에 인덱스 정의부터 작성
4. 클라이언트에서 관리자 권한 쓰기 금지

## 6.4 외부 API 연동 규칙

1. 타임아웃/재시도/서킷브레이커 정책 명시
2. 장애 시 graceful fallback 제공
3. 외부 응답 원문 로그 저장 금지(필요 시 마스킹)

---

## 7. 테스트 및 품질 게이트

## 7.1 테스트 피라미드

1. Unit: 도메인 로직
2. Integration: API + Firestore emulator
3. E2E: 핵심 사용자 여정

## 7.2 품질 게이트(머지 차단 조건)

1. lint/typecheck/test 모두 통과
2. 신규 API는 통합 테스트 최소 1개 이상
3. 보안 규칙 변경 시 시나리오 테스트 필수
4. 결제/AI 관련 변경은 회귀 테스트 필수

## 7.3 수동 QA 최소 항목

1. 회원가입/로그인/로그아웃
2. 만 14세 미만 동의 플로우
3. 검사 시작-중단-재개-완료
4. 학생부 기록 CRUD
5. 구독 결제/해지/실패 복구
6. AI 출력 금지표현 필터

---

## 8. 보안/개인정보 규칙

## 8.1 비밀정보 관리

1. 비밀키는 환경변수로만 관리
2. 키 노출 시 즉시 폐기 및 교체
3. 로그/에러 메시지에 민감정보 포함 금지

## 8.2 개인정보 최소수집

1. 서비스 목적과 무관한 정보 수집 금지
2. 필수/선택 항목 명확 분리
3. 파기 시점과 파기 방법 문서화

## 8.3 청소년 보호

1. 연령 검증 로직 단일화
2. 보호자 동의 전 기능 제한 정책 준수
3. 동의 기록 감사 추적 가능 상태 유지

---

## 9. AI 기능 운영 규칙

## 9.1 생성물 책임 경계

1. AI 결과는 초안으로만 제공
2. 사용자 검토 고지 문구를 UI에 고정
3. 허위 사실 생성 방지 정책 위반 시 출력 차단

## 9.2 프롬프트 관리

1. 시스템 프롬프트는 버전 관리
2. 프롬프트 변경은 `docs` 변경과 동시 반영
3. 성능/비용 영향 측정 후 배포

## 9.3 비용 통제

1. 기능별 토큰 상한
2. 사용자별 호출 제한
3. 월간 비용 임계치 초과 시 알림 및 제한

---

## 10. 결제/정산 운영 규칙

1. 웹훅 서명 검증 실패 요청은 전부 거부
2. 결제 이벤트는 멱등 처리
3. 결제 상태 변경은 이벤트 로그와 원장 데이터 동시 기록
4. 수동 환불은 운영 로그와 사유를 반드시 남긴다

---

## 11. 배포/릴리스 규칙

## 11.1 환경 분리

1. `dev`, `staging`, `prod` 분리
2. 각 환경별 Firebase 프로젝트 분리 권장
3. 운영 데이터 접근은 최소 권한 원칙 적용

## 11.2 배포 체크

1. 환경변수 누락 검사
2. 마이그레이션/인덱스 반영 상태 확인
3. 릴리스 노트 작성
4. 롤백 명령/절차 확인

## 11.3 롤백 규칙

1. 장애 등급 Sev1 발생 시 즉시 이전 안정 버전 롤백
2. 롤백 후 원인 분석 문서 24시간 내 작성

---

## 12. 운영/장애 대응 규칙

## 12.1 장애 등급

1. Sev1: 로그인/결제/핵심 데이터 손상
2. Sev2: 일부 기능 불가, 우회 가능
3. Sev3: 경미한 기능 저하

## 12.2 대응 시간 목표

1. Sev1: 15분 내 인지, 60분 내 임시 복구
2. Sev2: 60분 내 인지, 당일 내 복구
3. Sev3: 다음 배포 주기 내 처리

## 12.3 사후 분석

모든 Sev1/Sev2는 Postmortem 필수:

1. 타임라인
2. 근본 원인
3. 재발 방지 액션
4. 담당/기한

---

## 13. Definition of Ready / Definition of Done

## 13.1 DoR (개발 시작 조건)

1. 요구사항 ID 존재
2. 화면/API/데이터 영향 범위 명시
3. 수용 기준(AC) 정의
4. 테스트 관점 정의

## 13.2 DoD (완료 조건)

1. 코드/문서/테스트 동시 업데이트
2. 품질 게이트 통과
3. 운영 영향(로그/모니터링/알림) 반영
4. 릴리스 노트 반영

---

## 14. 변경 관리 규칙

1. 구조적 변경은 작은 단위로 분할 배포
2. 파괴적 변경은 deprecation 기간 제공
3. 문서 버전과 코드 변경을 같은 PR 또는 연결된 PR로 관리
