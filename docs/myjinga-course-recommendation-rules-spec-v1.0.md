# 마진가(Myjinga) 과목 추천 통합 규칙 스펙

- 문서 버전: v1.0
- 기준 문서: `docs/myjinga-development-spec-v1.1.md`, `docs/myjinga-project-rules-v1.0.md`, `docs/myjinga-curriculum-reference-data-spec-v1.0.md`, `docs/myjinga-course-planning-eligibility-spec-v1.0.md`
- 작성일: 2026-04-10
- 상태: Draft (추천 규칙 통합 기준)
- 적용 범위: 고등학교 보통 교과 기준 선택과목 추천, 이수 가능 여부 검사, 학교별 개설 필터링, 학기 배치 제약, 전공 적합도, 학습 부담도 반영 기준
- 영향 범위: 추천 API, 과목 설계 UI, 참조 데이터 모델, 검증 로직, 학교별 데이터 구조, 추천 설명 문구, 관련 개발 문서

## 1. 목적

본 문서는 현재 흩어져 있는 `이수 규정`, `학점 편성 규칙`, `과목/학과 매핑`, `학교별 개설 정보`, `학습 부담도` 규칙을 하나의 추천 기준 문서로 통합한다.

이 문서의 목적은 아래 5가지다.

1. 과목 추천에서 고려해야 하는 규칙의 우선순위를 고정한다.
2. `무엇을 들어야 하는가`, `몇 학점으로 들을 수 있는가`, `우리 학교에서 열리는가`, `언제 들을 수 있는가`, `학생에게 무리가 없는가`를 서로 다른 레이어로 분리한다.
3. 추천 엔진이 `하드 제약`과 `점수화 신호`를 혼동하지 않도록 기준을 명시한다.
4. 현재 데이터가 있는 규칙과 아직 없는 규칙을 구분해 단계적 구현 기준을 만든다.
5. 이후 학교별 로드맵 추천 기능의 단일 상위 기준 문서로 사용한다.
6. 목표 학교/학과가 정해진 이후 `목표 학과 권장 과목을 우선 배정`하고, 남은 학점에 대해 추가 추천하는 기본 플로우를 고정한다.

---

## 2. 범위와 전제

### 2.1 현재 범위

1. 대상 학교급: `고등학교 보통 교과`
2. 대상 입학생: `2025학년도 입학생`
3. 적용 교육과정: `2022 개정 교육과정`
4. 현재 정리 완료 범위:
  - `국어`, `수학`, `영어`, `사회(역사/도덕 포함)`, `과학` 선택과목 draft
  - 총 이수 학점 및 교과군별 필수 이수 규정
  - 공통/선택 과목 기본 학점 및 편성·운영 범위
  - 목표 학과 권장 과목 우선 배정 규칙 초안
5. 현재 미완료 범위:
  - 학교별 실제 개설 과목
  - 학교별 개설 학기
  - 과목 난이도/학습 피로도 프로파일

### 2.2 핵심 전제

1. 추천 엔진은 `전공 적합도`보다 먼저 `이수 가능성`을 확인해야 한다.
2. 학교별 실제 개설 정보가 없으면 일반 추천은 가능하지만, 최종 확정 추천은 `needsReview` 상태로 남겨야 한다.
3. 같은 과목이라도 학교별로 편성 학점과 개설 학기가 달라질 수 있으므로, 교육과정 규정과 학교 운영 정보를 분리해야 한다.
4. 학습 피로도와 난이도는 학생별 선호/역량 차이가 크므로, 기본값은 `soft penalty`로만 사용한다.
5. 학생이 목표 학과를 정하면 그 학과의 권장 선택과목은 `필수 배정 후보`로 먼저 잠근다.
6. 현재 학과별 권장 선택과목의 기본 출처는 `서울고교학점제지원센터`이며, 추후 학교별 공식 발표자료가 확보되면 우선순위를 재정의한다.

## 3. 규칙 축

본 서비스의 과목 추천은 아래 5개 축을 동시에 고려해야 한다.

| 축 | 질문 | 규칙 성격 | 현재 상태 |
| --- | --- | --- | --- |
| `이수 규정` | 졸업 요건을 충족하는가 | 하드 제약 | 정리됨 |
| `전공 적합성` | 학과/직업과 얼마나 맞는가 | 점수 신호 | 부분 정리 |
| `학교 개설 여부` | 우리 학교에서 실제로 열리는가 | 하드 제약 또는 강한 필터 | 미정리 |
| `과목 개설 학기` | 지금 또는 다음 학기에 들을 수 있는가 | 하드 제약 또는 강한 필터 | 미정리 |
| `학습 피로도/난이도` | 현재 계획과 함께 감당 가능한가 | 소프트 제약 | 미정리 |

## 4. 규칙 우선순위

### 4.1 기본 추천 플로우

본 서비스의 기본 플로우는 아래 순서를 따른다.

1. 학생이 검사 결과, 관심 학과 탐색, 사용자 입력을 통해 `목표 학교/목표 학과`를 정한다.
2. 목표 학과가 정해지면 해당 학과 진학에 유리한 `권장 선택과목 묶음`을 먼저 불러온다.
3. 이 권장 과목 묶음은 `필수 고정 과목(lockedRequiredSubjects)`으로 확정하고 먼저 계획에 넣는다.
4. 이 필수 고정 과목을 기준으로 이수 규정, 학교 개설 여부, 개설 학기 충돌을 검사하되 과목을 자동 제거하지 않는다.
5. 필수 배정 이후 남는 학점에 대해 전공 적합성, 균형, 학습 부담도를 고려해 추가 과목을 추천한다.
6. 최종 출력은 `1-1`, `1-2`, `2-1`, `2-2`, `3-1`, `3-2` 단위의 과목 로드맵이다.

### 4.2 목표 학과 권장 과목의 현재 운영 기준

1. 현재 단계에서는 `서울고교학점제지원센터` 학과 안내 정보를 목표 학과 권장 과목의 기본 출처로 사용한다.
2. 추후 학교별 공식 발표자료가 확보되면 출처 우선순위는 아래 순서를 기본으로 한다.
  - 학교별 공식 발표자료
  - 서울고교학점제지원센터
  - 내부 수동 검수 데이터
3. 목표 학과 권장 과목은 추천 점수 경쟁 대상이 아니라 `반드시 유지해야 하는 고정 배정 과목`으로 취급한다.
4. 학교 미개설, 학기 미개설, 총 이수 구조 충돌이 생겨도 조용히 제거하지 않고 `충돌 경고`를 노출한다.
5. 충돌이 해결되지 않은 상태의 로드맵은 `확정 추천안`이 아니라 `재검토 필요 로드맵`으로 분류한다.

추천 엔진은 아래 순서를 반드시 따라야 한다.

1. `목표 학과 권장 과목 우선 배정`
2. `이수 규정`
3. `학교 개설 여부`
4. `과목 개설 학기`
5. `전공 적합성`
6. `학습 피로도/난이도`

우선순위 원칙은 아래와 같다.

1. 앞 단계에서 탈락한 후보는 뒤 단계에서 점수가 높아도 복구하지 않는다.
2. `하드 제약`은 추천 후보를 제거한다.
3. `소프트 제약`은 후보를 제거하지 않고 점수를 가감한다.
4. 학교 데이터가 없을 때는 `제거`보다 `needsReview` 경고를 우선한다.

## 5. 축별 상세 규칙

### 5.1 목표 학과 권장 과목 우선 배정

#### 역할

목표 학과가 정해졌을 때, 해당 학과와 직접 연결된 권장 선택과목을 추천 엔진의 시작점이자 `필수 고정 과목`으로 확정한다.

#### 적용 규칙

1. 학생이 목표 학과를 선택하면 해당 학과의 권장 선택과목을 먼저 조회한다.
2. 이 과목들은 `lockedRequiredSubjects`로 등록하고 `필수 고정 과목`으로 취급한다.
3. `lockedRequiredSubjects`는 다른 일반 추천 점수보다 항상 우선하며, 일반 추천 단계에서 제거할 수 없다.
4. 필수 고정 과목이 학교 미개설, 학기 미개설, 이수 구조 충돌 상태라면 제거하지 않고 `conflictRequiredSubject` 상태로 남긴다.
5. 필수 고정 과목 충돌은 대체 과목으로 자동 치환하지 않고, 전체 계획을 `needsReview` 이상으로 승격한다.
6. 최종 화면에서는 `필수 고정 과목`과 `추가 추천 과목`을 구분해 보여준다.

#### 데이터 소스

1. 현재: `서울고교학점제지원센터` 기반 학과 안내 데이터
2. 추후: 학교별 공식 발표자료
3. 보조: `majors`, `majorSubjectLinks`

#### 엔진 반영 방식

1. `lockedRequiredSubjects`를 먼저 6개 학기 로드맵에 배치 시도
2. 충돌 시 대체 과목으로 자동 치환하지 않고 경고를 반환하며, 과목 자체는 유지한다.
3. 잔여 학점 추천은 이 단계가 끝난 뒤 수행한다.
4. 결과 화면에는 `고정 과목 배정 결과`, `충돌 과목`, `잔여 학점 추천 과목`을 함께 표시한다.

### 5.2 이수 규정

#### 역할

학생 계획이 교육과정 졸업 요건을 충족하는지 확인한다.

#### 적용 규칙

1. `총 이수 학점 192`
2. `필수 이수 학점 84`
3. `자율 이수 학점 90`
4. `창의적 체험활동 18학점(288시간)`
5. 교과군별 필수 이수 학점
  - `국어 8`
  - `수학 8`
  - `영어 8`
  - `사회(한국사 6 + 통합사회 8)`
  - `과학 10`
  - `체육 10`
  - `예술 10`
  - `기술·가정/정보/제2외국어/한문/교양 16`

#### 데이터 소스

1. `graduationRequirements`
2. `creditOnlySubjectGroupRequirements`

#### 엔진 반영 방식

1. 충족 불가능한 경우 `ineligible`
2. 현재 계획만으로는 부족하지만 향후 보완 가능한 경우 `needsReview`
3. 전공 점수 계산보다 먼저 적용

### 5.3 전공 적합성

#### 역할

`lockedRequiredSubjects`를 제외한 잔여 학점 구간에서, 학생이 관심 있는 학과/직업/검사 결과와 과목의 관련도를 계산한다.

#### 적용 규칙

1. `majorSubjectLinks`의 추천 강도(`core`, `recommended`, `exploratory`)를 기본 점수로 사용
2. `subjects.relatedMajorNames`는 보조 신호로만 사용
3. 관심 직업만 있는 경우 직업-학과 매핑 또는 검사 결과를 통해 학과 후보를 먼저 만든 뒤 과목을 연결
4. 같은 학과에 대해 여러 과목이 연결되면 `핵심 과목`을 우선 추천
5. 목표 학과 기반 필수 배정 과목은 본 단계의 경쟁 대상에서 제외한다.

#### 데이터 소스

1. `subjects`
2. `majors`
3. `majorSubjectLinks`
4. 추후 `jobMajorLinks`

#### 엔진 반영 방식

1. 하드 필터를 통과한 후보에 점수 부여
2. 설명 문구에는 항상 근거 학과/직업명을 같이 반환

### 5.4 학교 개설 여부

#### 역할

추천 과목이 학생 학교에서 실제 개설되는지 확인한다.

#### 적용 규칙

1. 학교에서 미개설된 과목은 최종 추천안에서 제외한다.
2. 학교 정보가 없으면 일반 추천은 허용하되 `개설 여부 미확인` 경고를 반환한다.
3. 학교가 동일 과목을 여러 학점 버전으로 운영하면 학교 데이터를 우선한다.

#### 데이터 소스

1. 추후 `schoolSubjectOfferings`
2. 보조로 `creditOperationPolicies`

#### 엔진 반영 방식

1. 학교 데이터 있음: 하드 필터
2. 학교 데이터 없음: `needsReview`

### 5.5 과목 개설 학기

#### 역할

추천 과목이 학생의 현재 학년/학기 계획 안에서 배치 가능한지 확인한다.

#### 적용 규칙

1. 현재 또는 다음 계획 학기에 개설되지 않으면 해당 학기 추천안에서는 제외한다.
2. 3개년 로드맵 추천에서는 미래 학기 개설 정보가 있으면 보류 후보로 둘 수 있다.
3. 학기 정보가 없으면 `학기 미확인` 경고를 반환한다.

#### 데이터 소스

1. 추후 `schoolSubjectOfferings.offeredTerms`
2. 추후 학교별 교육과정 편성표

#### 엔진 반영 방식

1. 단기 추천(이번 학기/다음 학기): 강한 필터
2. 장기 추천(3개년 로드맵): 보류 후보 허용

### 5.6 학습 피로도/난이도

#### 역할

학생의 한 학기 계획이 과도하게 어렵거나 편중되지 않도록 보정한다.

#### 적용 규칙

1. 난이도는 과목 절대 난이도와 학생 상대 적합도를 구분한다.
2. 동일 학기에 고난도 과목이 과도하게 몰리면 감점한다.
3. 읽기·쓰기 중심 과목, 수리 고난도 과목, 실험/탐구 과목의 균형을 본다.
4. 학습 피로도는 추천 후보를 제거하지 않고 우선순위를 조정하는 데 사용한다.

#### 데이터 소스

1. 추후 `subjectDifficultyProfiles`
2. 추후 학생별 `learningPreferenceProfile`
3. 보조로 `subjects.keyTopics`, `selectionType`, `subjectGroup`

#### 엔진 반영 방식

1. 기본값은 `soft penalty`
2. 데이터가 없으면 점수 조정 없이 설명만 생략

## 6. 데이터 엔티티 맵

### 6.1 현재 정리된 엔티티

1. `subjects`
2. `majors`
3. `majorSubjectLinks`
4. `graduationRequirements`
5. `creditOnlySubjectGroupRequirements`
6. `creditOperationPolicies`

### 6.2 추가가 필요한 엔티티

1. `schoolSubjectOfferings`
  - 학교별 실제 개설 과목
  - 권장 필드:
    - `schoolId`
    - `schoolName`
    - `schoolType`
    - `academicYear`
    - `subjectId`
    - `subjectName`
    - `offeredTerms: string[]`
    - `credits`
    - `status: "open" | "planned" | "closed"`
2. `subjectDifficultyProfiles`
  - 과목 난이도 및 학습 부담도 기준
  - 권장 필드:
    - `subjectId`
    - `difficultyLevel: 1..5`
    - `workloadLevel: 1..5`
    - `assessmentIntensity: 1..5`
    - `readingLoad`, `writingLoad`, `problemSolvingLoad`, `projectLoad`
3. `studentPlanContext`
  - 현재 학년/학기, 완료 과목, 계획 과목, 관심 학과/직업, 선호도
4. `majorGuideRecommendations`
  - 목표 학과 권장 과목 묶음
  - 권장 필드:
    - `majorId`
    - `sourceProvider`
    - `requiredSubjectIds`
    - `recommendedSubjectIds`
    - `sourceUrl`

---

## 7. 통합 추천 프로세스

### 7.1 입력

1. 학생 기본 정보
2. 완료 과목 및 계획 과목
3. 관심 학과/직업/검사 결과
4. 학교 정보(선택 시)
5. 목표 학교/목표 학과

### 7.2 처리 순서

1. 입력 무결성 검사
2. 목표 학과 기준 `lockedRequiredSubjects` 생성
3. `lockedRequiredSubjects`를 `1-1`부터 `3-2`까지 학기 로드맵에 우선 배치 시도
4. `graduationRequirements` 기반 이수 가능 여부 검사
5. `creditOperationPolicies` 기반 학점 편성 가능 범위 확인
6. `schoolSubjectOfferings` 기반 실제 개설 여부 필터
7. `offeredTerms` 기반 현재/다음 학기 배치 가능 여부 필터
8. 남은 학점에 대해 `majorSubjectLinks` 기반 전공 적합도 점수 계산
9. `subjectDifficultyProfiles` 기반 학습 피로도 보정
10. `1-1`부터 `3-2`까지 학기별 로드맵 생성
11. 추천안과 경고 사유 코드 반환

### 7.3 반환 원칙

1. `ineligible`
  - 이수 규정상 불가능
2. `needsReview`
  - 학교 개설 여부, 학기 정보, 피로도 데이터가 부족
  - 목표 학과 필수 고정 과목과 학교 운영 현실이 충돌
3. `eligible`
  - 현재 데이터 기준으로 하드 제약과 강한 필터를 통과

## 8. 현재 구현 단계에서의 반영 기준

### 8.1 현재 바로 쓸 수 있는 규칙

1. `이수 규정`
2. `credit-only 교과군의 학점 총량 검증`
3. `과목-학과 연결 기반 전공 적합성`
4. `기본 학점/편성·운영 범위` 참조
5. 목표 학과가 정해졌을 때 권장 과목을 `필수 고정 과목`으로 취급하는 규칙
6. 테스트용 `majorGuideRecommendations` draft를 통한 `lockedRequiredSubjects` 생성

### 8.2 아직 보류해야 하는 규칙

1. 학교별 실제 개설 여부
2. 학교별 개설 학기
3. 과목 난이도/학습 피로도 프로파일
4. 서울고교학점제지원센터 학과 안내 데이터의 대량 정규화 적재와 `subject_seq` 정밀 매핑

### 8.3 현재 추천 상태 해석

1. 학교 데이터가 없으면 추천 결과는 기본적으로 `needsReview`
2. 목표 학과 권장 과목은 먼저 고정 배정하고, 충돌 시에도 삭제하지 않고 `충돌 경고`를 띄운다.
3. 전공 적합도가 높아도 이수 규정을 깨면 `ineligible`
4. 학습 피로도 데이터가 없으면 추천은 가능하되, 부담도 최적화는 적용하지 않는다.

## 9. 구현 우선순위

### Phase 1. 현재 데이터 기반 추천

1. `graduationRequirements`
2. `creditOnlySubjectGroupRequirements`
3. `subjects`, `majors`, `majorSubjectLinks`
4. `creditOperationPolicies`
5. 목표 학과 기준 `lockedRequiredSubjects` 생성 및 우선 배치 로직

### Phase 2. 학교별 현실성 반영

1. `schoolSubjectOfferings`
2. 개설 학기 필터
3. 학교 선택 UX
4. 서울고교학점제지원센터 학과 안내 정규화

### Phase 3. 계획 품질 고도화

1. `subjectDifficultyProfiles`
2. 학기별 부담 균형 로직
3. 3개년 로드맵 최적화
